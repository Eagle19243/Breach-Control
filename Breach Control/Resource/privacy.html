Silent Quadrant Breach Control Privacy Policy


Introduction

Silent Quadrant Breach Control (the "platform") exists solely because of data breaches.

This page details how the privacy of personal data is handled within Silent Quadrant Breach Control and what information is collected by Silent Quadrant or its business partners when you use the platform.

 

How Silent Quadrant Breach Control handles user-provided data

The only data Silent Quadrant Breach Control ever explicitly requests from users is email addresses. When someone subscribes to notifications, or searches an email address or password, that information is never passed to any third parties under any circumstances, except if required by law.



When you search for an email address

Searching for an email address only ever retrieves the address from local device storage then returns it in the response, the searched address is never anywhere outside of your local device. See the Logging section below for situations in which it may be implicitly stored. 

 

When you search for breached passwords

The one-time password search feature searches previous data breaches for the presence of a user-provided password. The password is hashed client-side with the SHA-1 algorithm then only the first 5 characters of the hash are sent to Have I Been Pwned (HIBP) per the Cloudflare k-anonymity implementation.  Silent Quadrant Breach Control never sees nor receives the original password nor enough information to discover what the original password is.  For obvious reasons, Silent Quadrant Breach Control cannot store passwords or offer a continuous monitoring service for password breach. By using the platform, you also agree to be bound by the HIBP Terms of Service and Privacy Policy.



Silent Quadrant Breach Control only uses SHA-1 hashes of the one-time password search. No identifying data about who the password belonged to is stored. 

  

When Silent Quadrant Breach Control will contact you

Notifications are only ever sent, via the mobile app, to individuals who enter their email address for monitoring. 

  

How Silent Quadrant Breach Control protects data

Security in Silent Quadrant Breach Control is handled by a "defense in depth" approach, that is the fact that the service employs many different layers of security; including, but not limited to:

 

-       - All data transmitted over the internet is done over HTTPS

-       - Cloudflare is used extensively to block potentially malicious requests

-       - Rate limits on APIs are implemented at both the code level and via Cloudflare

-       - Regular security scans are performed to identify code or configuration vulnerabilities

-       - Firewalls are employed to limit access to services running on Microsoft Azure

-       - Third party components are kept well-maintained (see OWASP's Using Components with Known Vulnerabilities)

-       - External client-side dependencies are embedded using subresource integrity hashes

-       - Controlling entity and hosting
